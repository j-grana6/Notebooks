"Stupid" fix 1: Dropping Missing Values¶

Recall, Approximately 17% of the values are 0. This can be either because missing data is coded as 0's. For now I am just going to drop those. In the future, there are several ways to better address this issue. They include:

  • Linear Interpolation (bad)
  • Interpolating by MSA or county data (bad)
  • Parametric Spatial Model (ok)
  • Preprocessing Spatial Neural Network (good)
  • Simultaneous interpolation/forecasting Spatial Neural Network (good)

But before we get fancy, let's just be dumb and drop missing data

"Stupid" Input Generation : Examine the autocorrelation to "optimally" determine sample size.¶

Since we have 18 years of data, one way to train a neural network would be to give it 17 previous years and ask it to predict the 18th. However, this gives us a number of trainning samples equal to the number of zipcodes. Also, it is unlikely that the percent change from 1994 to 1995 affects the percent change from 2011 to 2012. If we can determine when the autocorrelation goes to 0, we can increase our sample size.

Visually, it looks like there is a negative autocorrelation for the 1 period lag but after that, the autocorrelations appear to be distributed normally and centered around zero. The high variance is due to very small time scales in which to calculate the auto-correlation. Therefore, I am making the executive decision to run this first neural network using 2 lags (of course we will check this for robustness later).

So finally we can build the model. In the simples case, we assume that:

\begin{equation} y_{t} = f(y_{t-1}, y_{t-2}) \end{equation}

Note that in this case, there is no dependence on the zipcode! Therefore, we are implicitly assuming that the DGP is the same for all $i$ and only depends on time lags. This is of course, very bad. The linear model specifies:

\begin{equation} y_{t} = c + \rho_1 y_{t-1} + \rho_2 y_{t-2} \end{equation}

We now estimate this model on the full datset

The root MSE is  0.264764550353

This is --- for lack of better words --- terrible. It says that on average, our forecast is off by 25%. Not that we are 25% away from the truth but our forecast for the change in employment is 25% off. (In other words, we predict employment increases by 50% but it actually only increases by 25%). This is not surpising since, as I said all along, this model is stupid/dumb. However, it will be interesing to see how the Neural Network compares in even this very dumb setup.

One way we can control for this is by controlling for the size of the zipcode. For example, zipcodes with fewer people employed will have a much higher variance in changes in employement. This model does not take that into account. In other words, the "penalty" for predicting that a zip code with 2 employs grows to 4 when it actually grows to 3 employees is the same as the penalty for predicting that a zipcode with 2,000,000 employees grows to 4,000,000 when it actually only grows to 3,000,000. Continuing with the theme of this write-up (stupidity), I am going to rerun the regression when I only look at zipcodes with over (arbitrarily picked) 1000 employees in 1994. (Of course, a better way to do this would be with weights, but we are not that smart, yet). Dropping these small zipcodes as well as the drops earlier reduce the total percent of the labor force represented by about 15 percent.

Improvements¶

There are many improvements we can make. Succinctly, they are:

  • Interpolating missing values
  • Including zipcodes with a small number of employees
  • Include Spatial Relationships
  • Include "macro" variables (MSA, County and other employment data)

I would like to tackle these in reverse order.
