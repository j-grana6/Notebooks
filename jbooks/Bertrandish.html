The problem with the Cournot Game¶

We found some issues with the physical (and econmic) reality of the basic formulation of the event driven Cournot game. They include:

  1. When there is a demand shock, typically the quantity consumed changes because people buy less. Therefore the assumption that prices automatically adjust to keep consumption at a constant rate is not true. Or in other words, demand shocks yield inventory buildups.

  2. The idea of a message being an infinitesimal rate of production is physi- cally awkward at best and physically impossible at worse.

  3. Even under a constant demand regime, quantities in the market vary and that is a piece of information that firms use to set their quantity and price.

Bertrand-like game¶

The goal of this version of market competition is to bring the Cournot model closer to reality.

The players¶
  • The player's set a price by sending messages to the market.
The Market¶
  • The market places "orders" from each of the firms by sending messages to the firm.
  • The content of each message is an order quantity that is determined by the prices of both firms
  • The rate at which the market sends messages to the firms is fixed
  • The state of the market determines the demand function for each firm.
Profits¶

Player's earn profits every time thet receive an order. Profits are compute as their revenue minus an exogenous cost. We completely abstract away from prodction decisions, inventory buildup and production cost structure, though there is no reason this would have to hold in the future.

Formalization¶

  • For any internal state, the players can set a price in $[5, 10, 15, 20]$
  • Players can only change their price upon receiving an order. That is, there are no spontaneous transmissions, only stimulated ones.
  • The market's state is defined by a triple, $(s, p_1, p_2)$. $p_1$ and $p_2$ are the most recent prices the market has received from the players. $s$ can take one of 3 values, $[30, 50, 70]$ representing states of demand. The prices in the market determine the order quantity. More specifically, the order quantity received by player $i$ is given by $Q_i = s_m - p_i + .5p_j$ where $s_m$ is the state of the market. (Economically, these goods are substitutes, the coefficient on $p_j$ gives the degree of substitutability)
  • The state of the market evolves stochastically according to an underlying Poisson process.
  • The player's state consists of the most recent price they sent to the market as well as the price and order quantity of their last two messages received.

Some Counting¶

  • The market can be in any one of $4\times3\times3=36$ states.
  • The players can be in one of approx $4 \times (4 \times 4 \times 3) \times (4 \times 4 \times 3)=4096$ states
  • This yields $4^{4096}$ strategies for each player...uh-ho

Shrinking the game¶

Now, let's just assume that the player's state is the most recent order it received. (That is, a price and quantity combo):

  • Then the state space has size $4\times 4 \times 3=48$ and the players have $4^{48}$ total profiles.
  • The total number of joiunt profiles is approx $4^{96}$ which is approx $10^{57}$. This might still be too big but let's give it a whirl.

function receive_message_and_react(sft::Player, message::Message)
    #     This function will have a player receive a message and then emit
    #     a message and possibly change its state.  If the current state of the players
    #     and the received message has not yet occured for the player, the player will 
    #     pick a message uniformly random.  If the player has received the message in 
    #     his current state, he will do UCT.  
        
    #     Note, this function does NOT update the MCTS nodes, it only picks the player's
    #     action.

    #     Output is a message.  The Player object may also change its currentstate method
    if ~ sft.trembling
        action = sft.strategy[message, sft.currentstate]
        #sft.currentstate = action[2]
    else
        if rand() < sft.trembleprob
            action = (sample(sft.messages), sample(sft.statespace))
            #sft.currentstate = action[2]
        else
            action = sft.strategy[message, sft.currentstate]
            #sft.currentstate = action[2]
        end
    end
    return action
end;

function generate_random_strategy(messages_to_send::Array{Message, 1}, messages_to_receive::Array{Message, 1}, 
    statespace)
    strategy = Dict{(Message, Any), (Message, Any)}()
    for instate in statespace
        for m_in in messages_to_receive
            messageout = sample(messages_to_send)
            newstate = sample(statespace)
            strategy[(m_in, instate)] = (messageout, newstate)
        end
    end
    return strategy
end;

    ## DEFINE AND COLLECT ALL SPONTANEOUS EVENTS FOR THE MARKET
    market_spon_dict = Dict()

    market_state_full = Any[]

    for ix in 1:length(market_state_full)
        # First the spontaneous state change

        market_spon_dict[current_state] = Spont[] # Dict where keys are possible events
        for newstate in market_state_full
            if (newstate[2]==current_state[2]) & (newstate[3]==current_state[3]) & (newstate[1]!= current_state[1])

                push!(market_spon_dict[current_state], s_event)
            end
        end
        # Now the spontaneous message emission
        p1 = current_state[2]
        p2 = current_state[3]
        intercept = current_state[1]
        p1oq = intercept -p1 +.5*p2
        p2oq = intercept - p2 +.5*p1
        p1msg = Message("Market", "Player 1", (p1, p1oq), p1tomarket) # Orderes traverse quickly

        push!(market_spon_dict[current_state], s_eventp1)
        push!(market_spon_dict[current_state], s_eventp2)
        push!(marketmessages, p1msg)
        push!(marketmessages, p2msg)

    return net

function get_profits(net::SFTNet, message::Message, t::Float64)
    if message.receiver=="Player 1"
        p = exp(-net.players[1].discount*t)*(message.content[1]*message.content[2] - message.content[2] * net.players[1].cost)
        prof = [p, 0]
    else
        p = exp(-net.players[2].discount*t)*(message.content[1]*message.content[2] - message.content[2] * net.players[2].cost)
        prof = [0, p]
    end
    return prof
end;

function simulate(net::SFTNet, t_end::Int64)
    t = 0.
    total_profits = [0,0]
    possible_events = get_possible_events(net)
    while t < t_end
        #print(net.market.state, "\n")
        #print(total_profits[1], "\n")
        #print(possible_events, "\n \n")
        probs = WeightVec(Float64[e.rate for e in possible_events])
        event = sample(possible_events, probs)
        #print(event, "\n \n \n")
        # An event is either a spontaneous transm/transi or a message received
        if (typeof(event)==Message)
            ix = findfirst(net.messagesintransit, event)
            splice!(net.messagesintransit, ix) # Removes the messge from the network
            # If the event is the player receiving a message
            if event.sender == "Market"
                player = net.sftdict[event.receiver]
                # "player" is now the Player type        
                action = handle_event(event, net)
                new_messages = action[1]
                net.messagesintransit = [net.messagesintransit, new_messages]
                immediate_profits=get_profits(net, event, t)
            else
                handle_event(event, net)
                immediate_profits = [0,0]
            end
        else
            new_messages = handle_event(event, net)
            net.messagesintransit = [net.messagesintransit, new_messages]
            immediate_profits = [0,0]
        end
        possible_events = get_possible_events(net)
        timeincrement = get_time_increment(possible_events)
        #immediate_profits = get_profits(net, t, t+timeincrement)
        t += timeincrement
        total_profits += immediate_profits
        end
    return total_profits
end;

Pre-PGT experiments¶

Although the game continues to infinity, because of a positive discount rate, the expected utilities should converge. Let's take a look and see what expected utility looks like for a given joint profile and different lengths of simulation. In this example we simulate the network 5000 times to estimate expected utility.

Again, this has nothing to do with PGT. Since the game is of infinite horizon, these simulations are meant to determine how many time steps is "enough" so that the expected utility is "close" to the expected utility if the game were to run until infinity.

  • Market changes it's state at a rate of .25
  • Market spontaneously places orders at a rate of 1 to each player
  • Per unit cost is 1
  • Players can set price to $5,10,15,20$

Confidence Interval Based PGT¶

In [ ]:
pmap(PGT, [params, params, params, params, params, params, params, params])

1) Sample a joint strategy profile and compute the expected utility "almost exactly" (i.e. with a large number of runs).

2) For each alternative joint strategy profile, begin sampling the stochastic process to compute the expected utility.

3) (Suppose we are computing the intelligence for player 1's profile). After each sample of the network, compute the confidence interval of player 1's expected utility under his alternative strategy at a pre-specified significance level.

4) If the expected utility of player 1's "outerloop" strategy is outside of the confidence interval of player 1's expected utility under the alternative strategy, classify the alternative strategy as either better or worse than the outerloop strategy according to whether the outerloop strategy is above or below the confidence interval.

5) Set a predefined maximum number of simulations in case the outerloop expected utility is never outside of the confidence interval.

function is_worse(net::SFTNet, t_max::Int64, nruns::Int64, baseeu::Float64, player::Int64,
    confintmin=50, z=2.576)
    ## The player is the index to compare in the simulate output.
    runningeu = 0
    sse = 0
    s = 1
    running_sum_squares= 0
    while s < nruns
        utility = simulate(net, t_max)[player]
        e = utility - runningeu
        runningeu += e/s
        sse += e* (utility - runningeu)
        variance = sse/s
        s += 1
        if s >confintmin
            # confintmin large enough so that we can approximate the mean by a normal distribution, not t.
            conf_l = runningeu - z*(variance/s)^.5
            conf_u = runningeu + z*(variance/s)^.5
            if (baseeu < conf_l) | (baseeu>conf_u)
                s = nruns #Exit Loop
            end
        end
    end
    return runningeu
end;

Where to go¶

This convinces me that the code is running and doing what is expected. We also have the ability to change any parameter of the model. Now is when we need to drill down what specific types of experiments we would like to conduct...

Scrap¶

  • First we need to fix the player object

    • We want the player type to set a current strategy and have a container for all possible strategies

  • We need a function that simulates the network up to a prespecified number of time steps where player strategies are given.

  • We need a way to sample player strategies

    • Strategies are mapings from (input, state) to (output, state).
    • To get a full set of strategies, we need to know all possible messages a player can receive.
    • Need a function that takes as an input all SFTs in a game and outputs the pure strategy space for all players.

  • We need a way to compute expected utility for given strategies

    • This just requires simulating a bunch of times and averaging the output
    • Produce a graph that shows the convergence of expected utility vs the number of simulation
    • Produce a graph the shows the convergence of expected utility for a given number of time steps

  • Then we need a PGT algorithm

    • Once the above is implemented this should be simple:
      • Setup containers (~ 5 lines of code)
      • Sample a strategy profile for both players (two lines of code)
      • Call the EU function (1 line of code)
      • Sample alternatives for player 1 and compute EU (2 lines)
      • Sample alternatives for player 2 and compute EU (2 lines)
      • In the process, remember all EUs computed previously

  • If we can generate a list of all proiles for each player, any records (expected utilities, intelligences and such) should be a dictionary with indices as keys. That way we save memory instead of turning an entire dictionary into a string.
